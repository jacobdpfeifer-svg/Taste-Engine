# Taste Engine

A system that learns a person's **aesthetic taste from choices** (never verbal description),
represents it as **geometry** (a location + direction in an embedding space), and later applies
it to **generate designs across mediums** — interiors, exteriors, apparel, color schemes.

Built to be handed to a Cursor agent. Start with `docs/TASKS.md`; the `.cursor/rules/` files
keep the build on-architecture.

## Why it's built "test-first"

The whole idea rests on one falsifiable claim: taste is a stable, measurable location, and
choices in some mediums predict choices in others. So the repo ships with that test already
implemented and passing on synthetic data (`backend/tests/test_geometry.py`). Phase 1's job is to
confirm it holds on **real** embeddings and **real** choices before anything else is built.

## Run it (works today, zero external models)

Backend (uses the deterministic `stub` embedding backend by default — no downloads):
```bash
cd backend
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
python scripts/generate_stimuli.py   # writes data/stimuli.json + placeholder SVGs (44/domain, 6 domains)
pytest                 # synthetic proof + a full simulated session through the API, both beat chance
uvicorn app.main:app --reload    # API on http://localhost:8000
```

Frontend:
```bash
cd frontend
npm install
npm run dev            # http://localhost:5173, proxies /api -> :8000
```

You'll get the real elicitation loop end to end: **images only** (no captions during elicitation),
pairs and odd-one-out triads, a domain picker (pick two mediums for a focused session), and a
dashboard that reports hold-out / cross-domain / ablation accuracy **with Wilson 95% confidence
intervals** so small-sample noise can't be over-read. The fitted kernel is saved to
`kernels/<person_id>.json`.

The placeholder SVGs encode three unmistakable style axes (warm/cool, minimal/ornate,
geometric/organic), so choosing by look produces a learnable signal even on the stub backend —
but a stub run only validates the plumbing. For the real Phase 1 gate:

```bash
# 1. put ~40+ photos per domain in folders (folder name = domain):
#    backend/data/stimuli/interior/*.jpg   backend/data/stimuli/apparel/*.png  ...
pip install open_clip_torch torch pillow
python scripts/ingest_images.py      # builds data/stimuli.json from the photo folders
python scripts/verify_openclip.py    # smoke-tests the real embedding backend (~600MB download)
EMBEDDING_BACKEND=openclip uvicorn app.main:app --reload
```


`docs/CONTEXT.md` records the project's origin, the research behind each design decision
(verbal overshadowing → choice-only elicitation; Bradley-Terry → pairwise fitting; Openness →
cross-domain triangulation), and the full source dossier.

## What this is when complete

A person sits down and, in a few minutes, does nothing but **pick between things** — pairs and
odd-one-outs spanning rooms, buildings, outfits, objects, art, palettes — never describing
anything. From those choices the system builds a **taste kernel**: a shared taste vector plus
per-domain offsets, a set of discovered-and-confirmed style axes (the AI names them, the person
ratifies), and a personalized color→emotion map. The kernel is validated against the person's own
held-out choices, so its confidence is measured, not asserted.

With that kernel, the person can then ask for design help in **any covered medium** and get an
on-taste, downloadable result: "design my living room," "how should I dress for a gallery
opening," "a palette that makes my office feel focused." Each output is generated by conditioning
an image/palette generator on the kernel, and ships with a plain-English rationale ("chosen
because your profile favors warm, uncluttered, organic") plus a warmer/colder control that feeds
back and sharpens the model over time.

The finished product is, in effect, a **portable model of one person's taste** — captured once
through pure choice, and reusable across every design decision in their life. The MVP in this repo
proves the part everything else depends on: that such a model can be learned and that it predicts.

See `docs/TASKS.md` for the phase-by-phase path from here to there, and the honest limits in
`docs/SPEC.md`.
